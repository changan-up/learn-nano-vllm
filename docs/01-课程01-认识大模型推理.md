## 核心概念

### LLM 推理在做什么

**训练**是更新参数；**推理**是固定参数下，根据输入 token 序列计算下一个 token 的分布，再采样，循环直到结束。推理不关心梯度，关心 **延迟、吞吐、显存占用**。

对 Decoder-only Transformer（如 GPT、Qwen、LLaMA）：

- 输入：一段 **prompt**（可视为 token 序列 \(x_1,\dots,x_T\)）。
- 输出：在 prompt 之后 **逐个生成** token \(y_1, y_2, \dots\)，直到 EOS 或达到最大长度。

### Prefill 阶段（填充阶段）

**定义**：第一次（或某一扩写轮次）对 **整个 prompt**（或一批新 token）做前向，计算每层的 **Q、K、V**，并写出 **KV Cache** 供后续步复用。

**特点**：

- **计算密集（Compute-bound）**：矩阵乘维度大，GPU 容易饱和算力。
- **可并行度高**：prompt 内 token 之间在因果掩码下仍可做高效注意力（FlashAttention 等）。
- 在 nano-vllm 中通常对应 `context.is_prefill == True` 的路径。

### Decode 阶段（解码阶段）

**定义**：从 **已生成的最后一个位置** 开始，每步只输入 **1 个新 token**（或每序列一步），读取 **历史 KV Cache**，计算当前步注意力，再采样下一 token。

**特点**：

- **内存/带宽密集（Memory-bound）**：每步算子相对小，瓶颈常在 **读 KV**、写 KV、以及内核启动。
- **步数多**：生成长度为 \(N\) 时，decode 要跑约 \(N\) 步（或按调度合并多序列）。
- 在 nano-vllm 中通常对应 `context.is_prefill == False`，并使用 `flash_attn_with_kvcache` 等 API。

### 自回归生成原理

**自回归**：第 \(t\) 个输出 token 的分布 **条件于** 前面所有 token（含 prompt 与已生成部分）。

\[
P(y_1,\dots,y_{L}) = \prod_{t=1}^{L} P(y_t \mid x, y_{<t})
\]

实现上就是 **for 循环**：每步 **前向** → **softmax 得 logits** → **采样或贪心** → 拼接到序列末尾 → 直到 EOS。

EOS详解：
EOS（End-of-Sequence Token） 是大语言模型（LLM）词汇表中预定义的特殊标记，用于明确标识文本序列的结束位置。在模型内部，EOS 以数字 ID形式存在（如 Llama 2 的2对应</s>），由分词器（Tokenizer）负责文本与 ID 的双向映射。
EOS 的工作机制（从训练到推理）
训练阶段
1.数据预处理：
分词器将文本转换为 token 序列
在每个完整样本末尾添加 EOS token
2.模型学习：
模型通过自监督学习，掌握 EOS 与语义完整性的关联
学习在合适的语法 / 语义边界生成 EOS，而非随意插入
3。损失计算：
EOS 与其他 token 一样参与交叉熵损失计算
模型学会预测 EOS 出现的正确位置
2. 推理阶段（核心流程）
初始化：输入prompt → 分词 → 生成初始token序列
循环：
  1. 前向传播计算下一个token的概率分布
  2. 应用采样策略（Greedy/Top-k/Top-p）选择token
  3. 检查是否为EOS或达到max_new_tokens
  4. 若是EOS：停止生成，返回结果（去除EOS）
  5. 若否：将token加入序列，更新KV缓存，继续循环

**为什么需要 KV Cache**：若每步都把 **整段历史** 重新算 K、V，复杂度会爆炸。缓存 **过去位置的 K、V**，新步只算 **当前 token 的 Q** 并与缓存 K、V 做注意力，是工业推理的标配。

### 为什么推理是瓶颈

1. **显存（GPU memory）**  
   - KV Cache 随 **序列长度 × 层数 × 头数 × head_dim** 增长，多用户并发时 **线性放大**。  
   - 激活、临时 buffer、权重（若未充分共享）也会挤占显存。

2. **计算（Compute）**  
   - **长 prompt** 的 Prefill 单次前向成本高；模型越大，FLOPs 越高。  
   - Decode 每步 FLOPs 可能不大，但 **总步数 × 并发** 仍可观。

   FLOP：Floating Point Operation → 一次浮点运算（浮点加、减、乘、除，都算 1 次）
   FLOPs：Floating Point Operations s → 浮点运算总次数（模型跑完一次要做多少次浮点计算）
   FLOPS（大写 S）：Floating Point Operations Per Second → 每秒浮点运算次数，硬件算力单位
   FLOPs（带 s 小写）= 总工作量（模型要算多少次）  FLOPS（大写 S）= 硬件速度（显卡每秒能算多少次）
   推理耗时 = 总工作量 ÷ 干活速度

4. **延迟（Latency）**  
   - 用户 **首 token 时间（TTFT）** 主要由 Prefill 决定。  
   - **每 token 时间** 在 Decode 阶段累积；长输出会拉长总延迟。

### 吞吐量 vs 延迟的权衡

| 指标 | 含义 | 典型关注点 |
|------|------|------------|
| **吞吐量** | 单位时间处理的 token 数或请求数 | 批大小、连续批处理、提高 GPU 利用率 |
| **延迟** | 单请求从进入到首 token / 每 token 的耗时 | 小批量、少排队、避免不必要的同步 |

**矛盾**：增大 batch 常提高吞吐，但可能让单请求等待更久（排队）；**连续批处理（continuous batching）** 试图在二者间取平衡——这正是 nano-vllm / vLLM 调度器的核心故事之一。

详解：
吞吐 Throughput：单位时间能处理多少个请求（追求跑得多）
单请求时延：一个请求从发出去 → 拿到完整回复的总耗时 = 排队等待时间 + 推理计算时间

增大 Batch 能提高吞吐
核心：分摊 GPU 固定开销，把 GPU 吃满
GPU 跑推理有固定开销算子启动、显存读写、内核调度、通信开销……不管你跑 1 个请求还是 8 个请求，这部分固定成本几乎一样。
小 Batch 浪费算力Batch=1：固定开销占比极高，GPU 大半时间在空转，算力利用率低。
大 Batch 摊薄开销一次塞进 16/32 个请求，固定开销只花一次，一次性算完一批。总耗时远小于「逐个单独跑」的累加耗时，单位时间处理请求变多 → 吞吐暴涨。

Batch 一大，单请求排队更久：
核心原因就两点：要凑人、要等批次
1. 必须凑满批次才开始推理（静态 Batch 最明显）
比如你设最大 Batch=16：
现在只来了 1 个请求
引擎不会立刻给你跑，而是等着后面再来 15 个凑满一批才开始计算
你这个请求就只能原地排队干等
Batch 设置越大，需要凑的请求越多，单个请求排队等待时间就越长。
2. 就算是 vLLM/nano-vllm 动态 Batch 也一样
动态不用严格凑满，但：
调大最大 Batch 上限后，调度器会倾向多攒一会儿请求，合并更大批次去跑，为了榨干 GPU 吞吐
请求不会来了就立刻调度，会短暂攒队，单请求等待时延自然上升
3. 大批次本身推理计算也变慢
就算不排队，同一个 GPU：
Batch=1 推理耗时：10ms
Batch=32 推理耗时：不会是 320ms，也会涨到 25～35ms
批次越大，矩阵乘法、KV 缓存调度压力越大，单批计算耗时变长，排在里面的每个请求都要多等。

### 与 nano-vllm 配置项的直觉对应（预告）

后续读 `config.py` 时，可把本课概念 **映射到数字**：

- **`max_model_len`**：单序列 **上下文长度上界**，直接影响 **KV 逻辑规模** 与调度可行性。
- **`max_num_batched_tokens`**：单步前向允许的 **token 总数上限**，是 **Prefill 长 prompt 批量合并** 时的硬约束。
- **`max_num_seqs`**：并发 **序列条数** 上界，与 **Decode 多请求同跑** 强相关。
- **`gpu_memory_utilization`**：给 **KV 块池** 等预留显存比例的「旋钮」，太大易 OOM，太小浪费显存。

这些不是本课推导重点，但你在回答「为什么推理慢 / OOM」时，把 **概念** 与 **配置名** 对上，会显得 **既懂算法又懂系统**。

## 源码解析（带完整源码和逐行注释）

本课是概念课，**不绑定具体 Python 文件**。下面给出一段 **伪代码**，对应你将来在 `llm_engine` 中看到的「外层循环」心智模型，便于与源码对照：

```python
# 伪代码：自回归 + prefill/decode 两阶段（非 nano-vllm 直接源码）

def generate_one(prompt_token_ids, max_new_tokens):
    # 1) Prefill：整段 prompt 一次前向，填充 KV cache
    logits = model.forward_prefill(prompt_token_ids)  # 计算密集
    next_token = sample(logits[-1])

    out = [next_token]
    # 2) Decode：每步只喂入一个新 token，读取 KV cache
    for _ in range(max_new_tokens - 1):
        logits = model.forward_decode([next_token])   # 内存带宽敏感
        next_token = sample(logits[0])
        if next_token == EOS:
            break
        out.append(next_token)
    return out
```

逐行说明：

1. **`forward_prefill`**：输入长序列，并行算各 token 的表示，**写满 KV cache**。
2. **`logits[-1]`**：通常只关心 **最后一个位置** 的预测作为第一个生成 token（与 `ParallelLMHead` 在 prefill 只取最后一个 hidden 一致）。
3. **`forward_decode`**：每步 **1 个 token**，主要依赖 **已缓存的 K、V**，故访存突出。
4. **循环与 EOS**：体现 **自回归**；真实引擎里还有 **batch 内多序列、不同长度**，由 `Scheduler` 管理。

## 图解（用文字/ASCII 描述）

**时间轴上的两阶段**：

```
Prompt tokens:  [t1][t2][t3]...[tT]
                      | Prefill 一次前向（算力为主）
                      v
KV cache 填满前缀位置
                      |
              采样得到 y1
                      | Decode 步 1（访存为主）
                      v
              采样得到 y2
                      ...
```

**单请求资源示意图**：

```
Prefill:  ████████████  (矩阵大，GPU 计算单元忙)
Decode:   ▁▁▁▁▁▁▁▁▁▁  (每步小，等内存、等 kernel)
```

## 面试考点

- **Prefill vs Decode** 的差异：**算力 vs 访存**、**输入长度**、**是否逐步增长序列**。
- **KV Cache 的作用** 与 **不算 cache 的复杂度问题**。
- **TTFT** 与 **per-token latency** 分别由什么主导。
- **吞吐 vs 延迟** 的矛盾及 **continuous batching** 的动机（点到为止，深讲在 Scheduler 课）。

## 常见面试题

1. **推理和训练反向传播的主要区别？**  
   答：推理不保留中间激活用于训练；优化目标从「收敛」转为 **延迟、吞吐、显存**；常 **用 KV cache、量化、算子融合**。

2. **为什么说 Decode 阶段更 memory-bound？**  
   答：每步 batch 内有效计算相对小，**KV 读取与写入** 占比高；同时 **kernel 启动次数多**，难以吃满算力。

3. **首 token 延迟主要受什么影响？**  
   答：主要受 **Prefill 长度与模型大小** 影响；多用户排队时还要加 **调度等待**。

4. **自回归能不能并行生成多个 token？**  
   答：标准自回归 **不能** 并行预测多个未来 token（因果性）；并行来自 **批内多请求** 或 **推测解码** 等扩展，本课不展开。

5. **Batch size 在 Prefill 与 Decode 中含义有何不同？**  
   答：**Prefill** 的 batch 常指「多少序列、多少 token 一起算」；**Decode** 常指「多少序列同步前进」，每序列每步通常 1 个新 token。调度器会用 `max_num_batched_tokens` 等限制 **总 token 预算**，避免 OOM。

6. **显存主要被谁吃掉？**  
   答：大模型推理中，**权重**（若未多卡切分）、**KV Cache**（随并发与长度增长）、**激活与临时 buffer** 是三大头；Serving 场景下 **KV** 往往最敏感。

## 小结

- **推理 = 固定权重下的前向 + 采样循环**；核心是 **Prefill（算）** 与 **Decode（访存）** 二阶段。
- **KV Cache** 是连接两阶段的桥梁，也是 **显存压力** 的主要来源。
- **吞吐与延迟** 需要在调度与批策略上权衡，这正是后续 **nano-vllm 引擎层** 要解决的问题。

**术语与 nano-vllm 对应（预习表）**：

| 概念 | 在仓库中的落点（后续课展开） |
|------|------------------------------|
| Prefill / Decode | `utils/context.py` 的 `is_prefill`；`scheduler` 分阶段 |
| KV Cache 布局 | `attention.py` 的 `k_cache`/`v_cache` + `store_kvcache` |
| 变长 batch | `cu_seqlens_q` / `cu_seqlens_k`（FlashAttention varlen） |
| 采样 | `layers/sampler.py` + `SamplingParams` |

**易错点提醒**：

- Prefill **不是**「只做 embedding」：而是 **整段 prompt 的完整 Transformer 前向**，并 **写入 KV**。
- Decode **不是**「不需要 KV」：恰恰相反，**强烈依赖** 历史 KV；每步只 **追加** 当前步的 K、V。
- **TTFT** 与 **总生成时间** 分开看：优化前者多从 **Prefill 与排队** 入手，优化后者常看 **Decode 步数与每步耗时**。
